# Analysis: Missed PCNT Activity During COOLDOWN

## The Problem
The log shows the CPAP machine generating SD bus activity at 15:05:15 while the firmware was in the `COOLDOWN` state. However, when the firmware later transitioned to `LISTENING` at 15:05:51, the activity was ignored, and the FSM remained suppressed until a *new* activity event occurred at 15:11:23.

## Root Cause
The `TrafficMonitor` correctly detected the activity during `COOLDOWN`. However, the FSM only evaluates if the bus is *currently* active (`trafficMonitor.isBusy()`) while calculating suppression inside the `handleListening()` function. 

Additionally, right before `COOLDOWN` transitions back to `LISTENING`, it explicitly calls `trafficMonitor.resetIdleTracking()`. This clears the pulse counter to ensure the next tracking cycle starts fresh. As a result, the transient activity that happened during `COOLDOWN` was completely wiped from memory before `handleListening()` ever got a chance to see it. 

We couldn't simply rely on removing `resetIdleTracking()`, because the ESP32's *own* SD card commands (during `UPLOADING` and `RELEASING`) generate bus activity that triggers the PCNT hardware. 

## The Solution: The Activity Latch
To solve this reliably without a major codebase overhaul, we introduced a boolean `_activityLatch` to the `TrafficMonitor`:

1. **Latching**: Anytime `TrafficMonitor` detects PCNT pulses, it permanently sets `_activityLatch = true`.
2. **Selective Clearing**: We added `trafficMonitor.clearActivityLatch()` to `main.cpp`, calling it *exactly* at the moment the ESP32 finishes an upload sequence and enters `COOLDOWN`. This safely wipes the "false" activity generated by the ESP32's own SD card writes.
3. **Capture**: From the precise moment `COOLDOWN` begins, the ESP is asleep, so any new bus activity is guaranteed to be from the CPAP machine. This activity will permanently throw the latch.
4. **Consumption**: Inside `handleListening()`, the FSM now checks `isBusy() || hasActivityLatch()`. If the latch was thrown during `COOLDOWN` (or any other state), `g_noWorkSuppressed` is immediately cleared and the latch is reset.

This guarantees that CPAP activity across *any* state is reliably captured and processed the moment the FSM is ready to evaluate suppression. The fix has been fully implemented.
